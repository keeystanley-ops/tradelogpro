

## **1. APPLICATION ARCHITECTURE SPECIFICATIONS**

### **1.1 Technology Stack Requirements**

**Frontend Framework:** Use Next.js version 14 or higher for React-based server-side rendering and static site generation capabilities. This ensures optimal performance and SEO capabilities.

**UI Library:** React version 18 or higher as the component framework for building interactive user interfaces.

**Styling Framework:** TailwindCSS version 3.4 or higher for utility-first CSS styling that enables rapid UI development with consistent design patterns.

**Component Library:** Implement shadcn/ui for pre-built accessible components that follow modern design standards and provide excellent user experience.

**Charting Libraries:** Combine Recharts for general data visualization with TradingView Lightweight Charts for professional trading-specific chart displays.

**State Management:** Use Zustand version 4 or higher for global state management due to its simplicity and performance benefits.

**Data Fetching:** Implement TanStack Query (React Query) version 5 or higher for server state caching, background updates, and optimized data synchronization.

**Form Handling:** Use React Hook Form combined with Zod for form handling and validation to ensure data integrity and excellent user input experiences.

**Backend Framework:** Node.js with Express version 20 or higher for the API server to handle all backend operations.

**Database:** PostgreSQL version 15 or higher as the primary data storage solution for relational data with strong consistency guarantees.

**ORM:** Prisma version 5 or higher as the database ORM for type-safe database queries and migrations.

**Caching Layer:** Redis version 7 or higher for session management and query caching to improve response times.

**Authentication:** NextAuth.js version 4 or higher for comprehensive authentication handling including OAuth for broker connections.

**File Storage:** AWS S3 or Cloudflare R2 for storing trade screenshots, exported reports, and user uploads.

**Real-time Communication:** Socket.io version 4 or higher for live P&L updates and real-time data synchronization.

**Hosting:** Deploy frontend on Vercel and backend on AWS or Railway for optimal performance and scalability.

### **1.2 Application Structure**

The application should follow a monorepo structure with separate applications for the web frontend, API backend, and background worker processes. Shared packages should contain common types, utilities, and UI components that are used across multiple applications.

The web application should organize pages into authentication flows (login, register, reset password), dashboard sections (dashboard, journal, trade log, reports, insights, notebook, playbook, challenges, trade replay, backtesting, settings), API routes, and shared components.

Components should be organized by feature area including UI components, charts, dashboard widgets, journal components, trade log components, reports components, navigation elements, and shared utilities.

The backend should follow a clean architecture with separate controllers, services, models, routes, middleware, and utility folders. Database schema should be defined in Prisma with proper migrations.

Background workers should handle broker integrations, trade imports, and export processes separately from the main application to ensure responsiveness.

## **3. DATABASE SCHEMA SPECIFICATIONS**

### **3.1 User Management Tables**

**Users Table:** Store user information including unique identifier, email address, password hash, first name, last name, avatar URL, timezone preference, currency preference, subscription plan tier, creation timestamp, and update timestamp. Email must be unique across all users.

**User Settings Table:** Store individual user preferences including reference to user, default broker selection, default risk amount, default risk percentage, breakeven threshold, default commission rate, real-time P&L display preference, dashboard layout configuration, chart settings configuration, and notification preferences. Each user should have exactly one settings record.

### **3.2 Broker Account Tables**

**Accounts Table:** Store broker account information including unique identifier, user reference, broker name, account number, account type (live or demo), currency, current balance, current equity, active status, last synchronization timestamp, encrypted credentials, creation timestamp, and update timestamp. Multiple accounts can belong to a single user.

### **3.3 Trade Tables**

**Trades Table:** Store all trade information including unique identifier, user reference, account reference, symbol, asset class, direction (long or short), strategy reference, entry date, entry price, entry time, quantity, entry commission, exit date, exit price, exit time, exit commission, realized P&L, realized R multiple, unrealized P&L, unrealized R multiple, maximum favorable excursion, maximum adverse excursion, stop loss price, take profit price, risk amount, risk percentage, risk reward ratio, status (open, closed, or partial), trade type (manual, imported, or synced), notes, rating from 1 to 5 stars, screenshot URLs, setup description, mistakes array, emotions array, time in trade in minutes, creation timestamp, and update timestamp.

**Trade Tags Table:** Store relationships between trades and tags including unique identifier, trade reference, tag reference, and category (setup, mistake, emotion, instrument, or custom). Each trade-tag combination should be unique.

**Tags Table:** Store user-defined tags including unique identifier, user reference, tag name, category, color hex code, and creation timestamp. Tag names should be unique per user.

### **3.4 Strategy Tables**

**Strategies Table:** Store trading strategies including unique identifier, user reference, strategy name, description, rules as JSON object, checklist as JSON object, active status, public visibility flag, creation timestamp, and update timestamp.

**Strategy Performance Table:** Store historical performance data for each strategy including unique identifier, strategy reference, date, win rate, profit factor, total trades, net P&L, and average R multiple. Each strategy-date combination should be unique.

### **3.5 Journal Tables**

**Journals Table:** Store journal entries including unique identifier, user reference, date, type (daily, weekly, monthly, or trade-review), title, content, mood indicator, rating from 1 to 5, associated trades, tags array, attachments array, creation timestamp, and update timestamp.

**Notebooks Table:** Store general notes including unique identifier, user reference, title, content, folder designation, tags array, pinned status, creation timestamp, and update timestamp.

### **3.6 Performance Tables**

**Performance Snapshots Table:** Store daily performance metrics including unique identifier, user reference, date, net P&L, win rate, profit factor, total trades, winning trades count, losing trades count, average win amount, average loss amount, largest win amount, largest loss amount, Zella Score, current streak, streak type, drawdown percentage, and equity. Each user-date combination should be unique.

### **3.7 Challenge Tables**

**Challenges Table:** Store trading challenges including unique identifier, user reference, challenge name, description, type (consistency, profitability, or discipline), target metrics as JSON, start date, end date, status (active, completed, or failed), progress percentage, and creation timestamp.

## **4. API ENDPOINT SPECIFICATIONS**

### **4.1 Authentication Endpoints**

**Register Endpoint:** Accept email, password, first name, last name, and timezone. Email must be valid format and unique. Password must be minimum 8 characters. Return user object, access token, and refresh token upon successful registration.

**Login Endpoint:** Accept email and password. Return user object, access token, and refresh token upon successful authentication. Implement rate limiting to prevent brute force attacks.

**Refresh Token Endpoint:** Accept refresh token and return new access token and refresh token pair. Implement token rotation for security.

**Logout Endpoint:** Accept authorization header with bearer token. Invalidate the token and return success confirmation.

**Forgot Password Endpoint:** Accept email address. Send password reset email if email exists in system. Return success confirmation regardless of email existence to prevent user enumeration.

**Reset Password Endpoint:** Accept reset token and new password. Validate token expiration and password requirements. Return success confirmation upon password update.

### **4.2 Dashboard Endpoints**

**Overview Endpoint:** Accept optional date range, account IDs, and asset class filters. Return net P&L, net P&L in R multiples, win rate percentage, profit factor, day win rate percentage, average win loss ratio, total trades count, winning trades count, losing trades count, Zella Score, current streak, streak type, maximum drawdown, and current equity.

**Daily P&L Endpoint:** Accept start date, end date, and granularity (day, week, or month). Return array of objects containing date, P&L amount, P&L in R multiples, trade count, and win rate for each period.

**Cumulative P&L Endpoint:** Accept start date and end date. Return array of objects containing date, cumulative P&L, cumulative P&L in R multiples, and equity for each date.

**Zella Score Endpoint:** Return overall score from 0 to 100, breakdown by category (profitability, consistency, risk management, discipline), and historical score data with dates.

### **4.3 Trade Endpoints**

**List Trades Endpoint:** Accept pagination parameters (page and limit), sorting parameters (sort field and sort order), and filtering parameters (status, symbol, asset class, strategy ID, tag IDs, date range, P&L range, direction). Return trades array and pagination metadata including page, limit, total count, and total pages.

**Get Trade Endpoint:** Accept trade ID. Return complete trade object including all related data (tags, notes, screenshots).

**Create Trade Endpoint:** Accept account ID, symbol, asset class, direction, entry date, entry price, quantity, stop loss, take profit, risk amount, strategy ID, tag IDs, and notes. Validate all required fields and calculate derived values (R multiple, risk reward ratio). Return created trade object.

**Update Trade Endpoint:** Accept trade ID and fields to update (exit price, exit date, stop loss, take profit, notes, rating, tag IDs). Recalculate P&L and related metrics. Return updated trade object.

**Delete Trade Endpoint:** Accept trade ID. Soft delete the trade or permanently remove based on configuration. Return success confirmation.

**Import Trades Endpoint:** Accept file upload (CSV or Excel), account ID, and column mapping configuration. Process file, validate data, create trades, and return count of imported trades, failed trades, and error messages.

**Bulk Update Trades Endpoint:** Accept array of trade IDs and update object. Apply updates to all specified trades. Return count of updated trades.

### **4.4 Reports Endpoints**

**Summary Report Endpoint:** Accept date range and group by parameter (day, week, month, symbol, or strategy). Return aggregated summary statistics.

**By Symbol Report Endpoint:** Accept date range. Return array of objects containing symbol, trade count, win rate, net P&L, average R multiple, and profit factor for each symbol.

**By Strategy Report Endpoint:** Accept date range. Return array of objects containing strategy ID, strategy name, trade count, win rate, net P&L, average R multiple, and profit factor for each strategy.

**By Tag Report Endpoint:** Accept date range and category filter (setup, mistake, or emotion). Return array of objects containing tag ID, tag name, trade count, win rate, net P&L, and average R multiple for each tag.

**By Time Report Endpoint:** Accept date range and granularity (hour, day of week, or month). Return array of objects containing time period, trade count, win rate, and net P&L for each period.

**Streaks Report Endpoint:** Accept date range. Return current streak length, current streak type, longest win streak, longest loss streak, and array of all streaks with type, length, start date, and end date.

**Drawdown Report Endpoint:** Accept date range. Return maximum drawdown, maximum drawdown date, current drawdown, and drawdown history with dates and equity values.

**Export Report Endpoint:** Accept date range, format (CSV, PDF, or Excel), and report type (summary, trades, or analysis). Generate file and return download URL with expiration timestamp.

### **4.5 Journal Endpoints**

**List Journals Endpoint:** Accept pagination parameters, type filter, and date range. Return journals array and pagination metadata.

**Get Journal Endpoint:** Accept journal ID. Return complete journal object including all related data.

**Create Journal Endpoint:** Accept date, type, title, content, mood, rating, trade IDs, and tags. Validate all required fields. Return created journal object.

**Update Journal Endpoint:** Accept journal ID and fields to update. Return updated journal object.

**Delete Journal Endpoint:** Accept journal ID. Return success confirmation.

### **4.6 Strategy Endpoints**

**List Strategies Endpoint:** Accept include public parameter. Return array of strategies belonging to user plus optionally public strategies.

**Get Strategy Endpoint:** Accept strategy ID. Return strategy object with performance history.

**Create Strategy Endpoint:** Accept name, description, rules object, checklist object, and public visibility flag. Return created strategy object.

**Update Strategy Endpoint:** Accept strategy ID and fields to update. Return updated strategy object.

**Delete Strategy Endpoint:** Accept strategy ID. Return success confirmation.

### **4.7 Tag Endpoints**

**List Tags Endpoint:** Accept optional category filter. Return array of tags belonging to user.

**Create Tag Endpoint:** Accept name, category, and color. Validate name uniqueness per user. Return created tag object.

**Update Tag Endpoint:** Accept tag ID and fields to update. Return updated tag object.

**Delete Tag Endpoint:** Accept tag ID. Return success confirmation.

### **4.8 Settings Endpoints**

**Get Settings Endpoint:** Return user settings object. Create default settings if none exist.

**Update Settings Endpoint:** Accept settings fields to update. Validate all values. Return updated settings object.

### **4.9 Broker Integration Endpoints**

**List Brokers Endpoint:** Return array of supported brokers with ID, name, logo URL, supported status, and authentication requirement flag.

**Connect Broker Endpoint:** Accept broker ID and credentials object. Authenticate with broker and return success status with connected accounts.

**Disconnect Broker Endpoint:** Accept broker ID and account ID. Remove credentials and return success confirmation.

**Sync Broker Endpoint:** Accept optional account IDs array. Sync trades from connected brokers and return count of synced trades, new trades, updated trades, and errors.

## **6. ZELLA SCORE ALGORITHM SPECIFICATIONS**

### **6.1 Score Calculation Methodology**

The Zella Score should be calculated as a weighted combination of four categories with an overall score ranging from 0 to 100.

**Profitability Category:** Weight of 35 percent. Calculate profit factor and allocate up to 15 points. Calculate average R multiple and allocate up to 10 points. Calculate win rate percentage and allocate up to 10 points. Normalize the sum to 35 points maximum.

**Consistency Category:** Weight of 25 percent. Calculate day win rate and allocate up to 10 points. Calculate streak score based on win and loss streaks and allocate up to 8 points. Calculate variance score based on P&L consistency and allocate up to 7 points. Normalize the sum to 25 points maximum.

**Risk Management Category:** Weight of 25 percent. Calculate average risk reward ratio and allocate up to 10 points. Calculate maximum drawdown and allocate up to 8 points with inverse scoring. Calculate position sizing consistency and allocate up to 7 points. Normalize the sum to 25 points maximum.

**Discipline Category:** Weight of 15 percent. Calculate plan adherence based on rule following and allocate up to 6 points. Calculate journal completeness based on note quality and frequency and allocate up to 5 points. Calculate rule following based on stop loss usage and risk limits and allocate up to 4 points. Normalize the sum to 15 points maximum.

### **6.2 Score Interpretation Guide**

**Excellent Range:** 90 to 100 indicates elite trader performance with strong profitability, consistency, risk management, and discipline.

**Very Good Range:** 80 to 89 indicates strong, consistent performance with minor areas for improvement.

**Good Range:** 70 to 79 indicates solid performance with room to improve in specific categories.

**Average Range:** 60 to 69 indicates typical trader performance with balanced strengths and weaknesses.

**Below Average Range:** 50 to 59 indicates needs improvement in key areas with specific recommendations provided.

**Poor Range:** 0 to 49 indicates significant issues to address with priority recommendations provided.

## **8. USER FLOW SPECIFICATIONS**

### **8.1 Onboarding Flow**

**Step 1 Sign Up:** User enters email and password. System validates email format and password requirements. User verifies email through confirmation link. User completes profile with name, timezone, and currency preferences.

**Step 2 Account Setup:** User optionally connects broker by selecting broker, authenticating, and selecting accounts to sync. Alternatively, user imports trades by uploading file, mapping columns, and reviewing and confirming. User can skip for manual entry.

**Step 3 Preferences:** User sets default risk settings including risk amount and risk percentage. User configures dashboard layout. User sets notification preferences.

**Step 4 Tutorial:** System shows platform overview. System walks through key features. System guides through first trade entry.

**Step 5 Dashboard:** User lands on dashboard with welcome message and empty state guidance if no trades exist.

### **8.2 Trade Entry Flow**

**Step 1 Initiate:** User clicks Add Trade button. Trade form modal opens.

**Step 2 Fill Trade Details:** User selects account from dropdown. User enters symbol with autocomplete suggestions. User selects direction using Long and Short toggle. User enters quantity. System optionally auto-calculates position size.

**Step 3 Entry Information:** User enters entry date and time. User enters entry price. System validates all entries.

**Step 4 Risk Management:** User enters stop loss and system auto-calculates R. User enters take profit and system auto-calculates Risk Reward. User enters risk amount or percent. System shows validation warnings.

**Step 5 Categorization:** User selects strategy from dropdown. User adds tags for setup, emotion, and mistakes. User adds notes.

**Step 6 Submit:** User can save trade, save and add another, or cancel. System confirms successful save.

### **8.3 Report Generation Flow**

**Step 1 Navigate:** User navigates to Reports section. System displays report categories.

**Step 2 Select Report Type:** User selects from Summary, By Symbol, By Strategy, By Tag, By Time, Streaks, or Drawdown reports.

**Step 3 Configure Filters:** User sets date range. User optionally filters by accounts, asset classes, symbols, strategies, and tags.

**Step 4 Generate Report:** System shows loading state. System displays results with charts and tables.

**Step 5 Export:** User optionally selects format from CSV, PDF, or Excel. User downloads file. User optionally shares link.

## **10. PERFORMANCE REQUIREMENTS**

### **10.1 Loading Time Targets**

**Initial Page Load:** Less than 2 seconds from navigation to fully interactive page.

**Dashboard:** Less than 1.5 seconds from navigation to fully rendered dashboard with all metrics.

**Trade Log:** Less than 2 seconds from navigation to fully loaded trade table.

**Reports:** Less than 3 seconds from navigation to fully generated report.

**Trade Detail Modal:** Less than 500 milliseconds from click to fully rendered modal.

**Chart Rendering:** Less than 1 second for all charts to render and become interactive.

### **10.2 Caching Strategy**

**Query Stale Time:** 5 minutes before data is considered stale and refetched in background.

**Query Cache Time:** 30 minutes before data is removed from cache.

**Refetch on Window Focus:** Enabled to ensure fresh data when user returns to tab.

**Retry Count:** 3 retries for failed queries with exponential backoff.

**Mutation Retry Count:** 1 retry for failed mutations.

**Cache Keys:** Should include user ID, date range, filters, and report type for proper invalidation.

### **10.3 Optimization Techniques**

**Virtual Scrolling:** Implement for large tables to render only visible rows.

**Chart Data Sampling:** Sample data points when exceeding maximum display points to maintain performance.

**Lazy Loading:** Implement for heavy components like Trade Detail Modal and Reports Page.

**Image Optimization:** Use WebP format, responsive images, lazy loading, and CDN delivery.

**Code Splitting:** Split code by route and feature to reduce initial bundle size.

**Service Worker:** Implement for offline support and faster repeat visits.

## **11. SECURITY REQUIREMENTS**

### **11.1 Authentication Security**

**JWT Configuration:** Use RS256 algorithm. Access token expiry should be 15 minutes. Refresh token expiry should be 7 days. Refresh token rotation should be enabled.

**Password Requirements:** Minimum 8 characters. Require uppercase letters. Require lowercase letters. Require numbers. Require special characters.

**Session Management:** Maximum 5 concurrent sessions per user. Idle timeout should be 30 minutes. Absolute timeout should be 24 hours.

### **11.2 Data Protection**

**Encryption at Rest:** Use AES-256 encryption for all stored data.

**Encryption in Transit:** Use TLS 1.3 for all network communication.

**Broker Credentials:** Use AES-256-GCM encryption for stored broker credentials.

**Rate Limiting:** Limit to 100 requests per minute per user.

**CORS:** Whitelist domains only.

**CSRF Protection:** Implement double-submit cookie pattern.

**Input Validation:** Use Zod schemas for all input validation.

**SQL Injection Prevention:** Use parameterized queries throughout.

**Audit Logging:** Log login attempts, data exports, settings changes, and trade modifications.
